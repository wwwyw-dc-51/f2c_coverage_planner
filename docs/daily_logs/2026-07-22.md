### 核心发现：补偿链

```
Sweep 分解 (根改动，替代 F2C Boustrophedon 网格)
  ├── 对多边形方向敏感 ──→ Phase 4A 倾斜 Sweep 补偿
  ├── 产生碎片条带 ────→ Cell 合并补偿
  ├── 斜边三角缝隙 ────→ 斜边检测 + 独立角度补偿
  ├── 全局角度不适用 ──→ per-cell veto 补偿
  └── veto 改变方向 ────→ 1e9 穿洞硬约束补偿
```

**本质**：Sweep 打破 F2C 原生假设 → 每个打破点需要补偿 → 补偿层互相耦合 → 改动任一点触发连锁反应。

同时 `robot_width` 有 6 重语义（headland 侵蚀/合并阈值/窄通道检测/min cell 面积/自适应 headland/边界补线偏移），`param-decouple` 实验证明存在未识别的隐藏依赖。

### 对比 Demo 算法
Demo 使用凹点递归切分 + 孔洞边界分割 + S1b 方向保护合并。核心优势：
- "只切带洞块"——无洞 block 不切割
- 方向保护合并——同时支持水平和垂直方向合并，5° 方向容差

结论：不换算法，借鉴 demo 两个原则改造 sweep。

### 迭代 1（898c0aa）：全局 X 切割 ❌

**做了什么**：在 `rectilinearDecompose` 的 sweep 模式中，给 `xs` 数组加入所有孔洞的 X 边界（minX, maxX），形成对齐孔洞的网格。

**想法**：X 边界让网格对齐孔洞列 → 子矩形更干净 → 合并更有效。

**结果**：
- N3: 98.1% → **100%** ✅
- N10: +11.8 分 ✅
- **S7: 100% → 98.42%（9→41 cells）❌**
- **S3: 100% → 99.15% ❌**
- **N13: 100% → 99.35% ❌**

**根因**：全局 X 切割对所有场景生效，不规则孔洞场景（S7/圆形障碍物）产生大量无法合并的碎片。合并 3 轮都无法吸收。

### 迭代 2（fd707fd）：条件触发 ❌

加了判定条件：存在 Y 重叠且 X 不重叠的孔洞对时才加 X 切割。

**结果**：S3 修好了（不触发），但 S7/N13 仍然退化——工厂场景孔洞多，条件仍然触发。

### 迭代 3（2022dcd）：更严格条件 ❌

条件改为 `≥2行×≥2列 或 单行≥3列`。

**老板反馈**："你这个会不会有点太面向测试场景编程了"

**认识到的错误**：

## 下午：正确实现

### 方案 2 正确实现（fa8b8d4）

**在哪里改**：`decomposer.cpp` 的 `rectilinearDecompose`，`cell.size() > 1` 分支

**之前**：含孔洞全宽条带 → `difference(孔洞)` → GDAL 产生不规则碎片

**之后**：
1. 收集该 cell 内所有孔洞的 X 边界（minX, maxX）
2. 排序去重，加上 cell 自身的 x0, x1
3. 在相邻 X 边界之间创建子矩形
4. 每个子矩形独立 `intersection(子矩形, 多边形)`
5. 无洞 → 干净矩形直接加入；仍有洞（非矩形孔洞边）→ 回退 difference

```
之前：全宽条带 + difference → [碎片A][洞][碎片B][洞][碎片C]
之后：子矩形逐个求交   → [矩形A][洞][矩形B][洞][矩形C]
      矩形A/B/C 是干净几何，不经过 GDAL difference
```

**关键**：X 切割只发生在有洞条带内部，不影响无洞条带（gap/入口/出口保持完整）。

## 关键教训

1. **方案驱动，不要自作主张加东西**。方案 1+2 都没提 X 切割，我却把它实现成了全局 X 切割。
2. **面向测试编程是坏味道**。加了三个越来越复杂的条件，只是为了通过特定场景——正确的做法是改进合并能力，不是给切割加门槛。
3. **分解和合并是一对**。分解产生什么形状，合并必须能吸收。方案 2 的子矩形拆分是正确方向——产生干净的矩形而不是 GDAL difference 的不规则碎片。
4. **第三轮合并效果存疑**。初步分析认为大部分场景是 no-op，需要实际 cell 数对比验证。
