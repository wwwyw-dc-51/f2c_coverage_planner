## v9.11+ — 边界补线距离感知 skip 修复

### 背景

S2 L形场景顶部 swath 离多边形边界太近（0.19m < robot_half_width=0.375m），机器人本体越界。
`fillBoundaryGaps` 的 skip 逻辑检测到已有 swath "覆盖"边界后直接跳过补线，不检查 swath 离边界的实际距离。

### 根因

`fillBoundaryGaps` 中 `boundary_already_covered` 判定只看 overlap 是否足够覆盖边界段，不检查 covering swath 的法向距离。当 swath 离边界 < robot_half_width 时，机器人越界但补线被 skip。

### 修复

**commit**: `a0e9f1d`

两处改动：

1. **`boundary_filler.cpp`** (line 529-535)：覆盖判定增加距离感知
```cpp
if (overlap >= seg_len - cov_width - 1e-6) {
    if (robot_half_width <= 0.0
        || max_normal_distance >= robot_half_width - 1e-6) {
        boundary_already_covered = true; break;
    }
    // swath 太近：不标记 covered，让边界补线触发
}
```

2. **`swath_generator.cpp`**（3 处调用）：`robot_half_width` 参数从 `0.0` 改为 `r_w * 0.5`

### 尝试过的方案（已废弃）

- **filterBoundarySwaths 预过滤**：删除太近 swath 再让 boundary fill 补偿 → 窄场景(S4)全删崩，覆盖丢失
- **原地替换（map 方案）**：边界补线替换原 swath 保持数量不变 → 位置变化仍影响 S7 routing，穿洞崩溃
- **孔洞 guard**：有孔洞场景跳过删除 → 老板指出大部分场景有孔洞，不可行

**教训**：任何改变 swath 位置/数量的操作都会影响复杂孔洞场景的 routing 拓扑。当前方案（只触发补线、不改原 swath）是唯一 8/8 通过的。

## v9.11++ — 消重 + 防斜边补线 + S7 脆弱性发现

### 背景

距离感知 skip 触发后，边界补线增多。S2 底部出现近重复补线，S7 出现 16.7° 斜边补线。

### 修复

**commit**: `703b44d` — merged to `main-stable`

1. **消重**：combined safety+coverage check。补线判定从单纯 overlap 改为联合判定（距离在 [robot_half_width, boundary_offset] 区间才标记 covered），消除近重复 fill。
2. **防斜边补线（外环）**：边界 fill segment 偏离 swath 主方向 >10° 则 skip。`cos(10°)=0.9848`。
3. **swath 数量**：53 → 52（去掉了 S2 底部的冗余 fill）

### S7 脆弱性发现

**任何改变 swath 数量/位置的改动，S7 一定穿洞崩溃。**

验证过程：
- 孔洞边界补线加 10° 角度检查 → S7 crash（少了一条 fill → routing 拓扑变化 → 穿洞）
- `adjustSwathEndpointsAdaptive` 端点缩进 → S7 crash
- 干净代码 stash 后 → S7 通过

**根因**：S7 10 个孔洞在 30×20m 区域内，cell decomposition 合并完只剩 2 个大 cell，routing 拓扑极其脆弱。这不是代码质量问题，是多边形+孔洞组合本身卡在临界状态。

**commit**: `e8ffcb3` (`feat/visibility-graph-path-planning`) — 保留 `adjustSwathEndpointsAdaptive` 死代码 scaffold，等 routing 更鲁棒后再启用。
