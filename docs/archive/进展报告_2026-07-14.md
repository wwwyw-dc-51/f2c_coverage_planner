## 1. 项目概述

基于 **Fields2Cover (F2C)** 库 + **ROS2 Humble** 的扫地机器人全覆盖路径规划系统。输入任意多边形区域（可含孔洞障碍物），输出完整的覆盖路径。

**机器人参数**: 宽度 0.95m，覆盖宽度 0.45m，差速驱动（原地转向）

### 07-10: cell 圆形绕孔洞排序

**问题**：F2C `genRoute` 的 TSP 求解器按欧氏距离排序 cell，在孔洞周围产生大量穿越连接。

**方案**：计算每个 cell 的 swaths 质心 → 按孔洞质心极角排序（顺时针） → cell 间连接走 headland 路径。TSP 优化限制在 cell 内的 swath 排序。

**尝试过但回退的方案**（来自日志）：
- 方向B bbox 重排：34 次交换但路径变长 + 穿越增加
- 策略3 跨组整组交换：692m→818m
- 2 组半平面分割：仍有穿越
- 手动 route 构建（Boustrophedon）：越界

当天也添加了 `fillBoundaryGaps` 但随即禁用——投影法 + 浮点阈值误触发填充，边界空隙问题未解决。

### 07-11 转折点：去重移除 + 评估修正

**fillBoundaryGaps 去重移除**：F2C 原生的去重逻辑把"顶在边界的合法边界补刀"误判为重叠跳过。移除该去重后，S4 窄走廊覆盖率 75%→97.4%（+22.4pp），S6 93.9%→94.3%。代码反而从 2808 行回退到 2764 行。

**评估器 3 项修复（v2）**：
1. `score_work_ratio` 未封顶 → `min(1.0, work_ratio)`（重叠 >100% 时得分失控）
2. `score_overlap` >100% 无区分度 → `exp(-overlap)` 替代 `1.0 - overlap_rate`
3. 网格法用侵蚀后 `no_hl` 作目标 → 改用全图 `cell`（headland 也算覆盖区域）

**首轮 6 场景基准（07-11，去重移除 + 修正评估器，几何法）**：
S1=98.59%, S2=98.63%, S3=95.95%, S4=97.44%, S5=98.21%, S6=94.26%

### 07-12 范式变革：Sweep 扫描线分解

自定义 `rectilinearDecompose(sweep_only=true)`：仅用孔洞顶点 y 坐标做水平切分线，生成全宽条带。S3: 42 cells→7（合并后 6），S6: 20 cells→大幅减少。合并角度阈值放宽到 60°。

关键决策：**sweep 模式下移除孔洞边界补刀**——盲补 swath 穿越孔洞，移除后覆盖率反而提升。新增显式 GDAL difference 处理小孔洞（如 S6 柜子）。

**网格评估加速**：Y 轴预过滤，O(G×P)→O(G×P_filtered)，加速 20-50x。

**6 场景 sweep 基准（07-12，网格法）**：
S1=98.60%, S2=98.68%, S3=97.91%, S4=95.00%, S5=99.04%, S6=98.56%

## 3. 核心算法突破（详解）

### 3.1 Sweep 扫描线分解 — 从 42 cells 到 7 cells

**问题**: 原生 Boustrophedon 将多边形沿短轴方向反复切分，在含孔洞的复杂场景（S3）产生 42 个 cell。大量 cell 间的 TSP 连接路径频繁穿越孔洞。

**方案**: 自定义 `rectilinearDecompose()`，仅沿水平方向按孔洞边界顶点切分，产生全宽条带式 cell。配合 sweep_only 模式，收集孔洞顶点 y 坐标作为切分线，x 方向覆盖多边形全宽。

**效果**: S3 场景 cell 数从 42 降至 7，连接路径复杂度大幅降低。条带式 cell 天然适合 Snake 排序。

### 3.2 斜边感知 Swath 角度（Phase 2A）

**问题**: sweep 分解产生水平条带 cell。当多边形有斜边界（如 notched 场景的 45° 斜面），固定水平 swath 会在斜边末端留下三角形未覆盖区域。

**方案**: `detectSlantedBoundaryAngle()` 检测 cell 是否贴近斜边界：
1. 收集 cell 外环所有线段，计算与 sweep 方向（≈水平）的夹角
2. 过滤：夹角 <15° 无意义（平行），>75° 也不采用（垂直边做水平 swath 即可）
3. 15°~75° 范围的斜边 → 用斜边方向作为该 cell 的独立 swath 角度

**创新点**: cell 级别自适应，不改变全局分解策略。垂直边被显式过滤，避免在水平条带中做荒谬的垂直 swath。

### 3.3 Snake 安全路径（Phase 2E）— 零穿洞保证

**问题**: v0-v2 都使用 F2C `genRoute`（基于 TSP 求解 cell 访问顺序），但其 Dubins/Reeds-Shepp 连接线在孔洞密集区域频繁穿越障碍物。S3 详细评估显示 66 次穿越。方案 A 事后修补治标不治本。

**方案**: Snake 模式完全跳过 TSP genRoute，手动构建 Route：
- Cells 按圆形绕洞排序（空间连续性优于 TSP 任意序）
- 逐 cell 拼接 swaths + 直连接（不经过 TSP 求解器）
- 空 cell 安全检查，跳过并找下一个非空 cell
- 同 cell 内 swaths 使用 F2C `SnakeOrder` 排序

**效果**: 穿越次数从概率性降低变为**确定性零穿越**。路径安全性从"大概率不穿"升级为"数学上不可能穿"。

### 3.4 RemArea 间隙填补（Phase 2C）

**问题**: 评估后仍有小面积未覆盖（网格 0.1m 下可见），传统做法需整个区域重新规划。

**方案**: 后置式修补，不修改已有路径：
1. 评估器输出 `uncovered_grid` 二值图
2. 空间分箱加速聚类（3×3 邻域 bin 搜索，O(n²)→O(n)，5000 点上采样限制）
3. 对聚类生成 bbox → 填充 swath → 裁剪后追加到路径

### 3.5 倾斜 Sweep（Phase 4A）

**问题**: sweep 固定沿水平方向分解，但多边形最长边可能不水平（如 notched 场景），导致 swath 方向非最优。

**方案**: `rotateCell()` / `rotateSwath()` 辅助函数。自动检测多边形最长边方向，若偏离水平 >10° 则将整个多边形旋转至水平 → sweep 分解 → 旋转回来。

### 3.6 边界容差放宽 + 斜边感知合并（Phase 2B + 2D）

**Phase 2B**: fillBoundaryGaps 角度容差从 cos5°(0.996) 放宽至 cos20°(0.9397)。更多斜边被正确识别为"与 swath 基本平行"，获得边界补填。

**Phase 2D**: 孔洞隔离检查新增"物理贴近"判定——若两 cell 边界距离 < 0.5×覆盖宽度，即使质心连线穿过孔洞也允许合并。防止 sweep 在斜边界产生孤立的三角 cell。

### 4.4 关键发现：Snake 模式的真实表现

**Snake 的"零穿洞保证"未兑现。** 日志显示 S3 两种模式的孔洞穿越情况：

| | S3 custom | S3 snake |
|------|:--:|:--:|
| 简化路径穿洞段数 | **0** | **6** |
| 详细路径穿洞段数 | 8 | 14 |
| 简化路径点数 | 285 | 275 |
| 网格法覆盖率 | 97.85% | 99.01% |

snake 跳过 TSP genRoute 后，cell 间用直连方式，这些直线穿越了孔洞区域。网格评估器将孔洞内靠近穿越段的网格点计为"已覆盖"，导致覆盖率虚高。**Snake 对比 custom 的覆盖率优势（+1.16pp）不能排除测量污染**。

这不是 snake 排序本身的问题（SnakeOrder 在 cell 内排序 swath 是合理的），而是 **cell 间连接方式** 的问题——跳过 TSP 的 headland 路由后，直连路径在孔洞密集场景无法保证安全。

## 5. 架构改进

### 5.1 评估体系

建立三层金字塔评估框架：
- **第一层**（硬门槛）: 覆盖率 ≥99.5% 合格，<95% 一票否决
- **第二层**（效率）: 路径长度、转弯次数、重叠率、规划耗时
- **第三层**（泛用性）: 跨场景通过率

### 5.2 参数体系

从最初的 ~12 个 ROS2 参数扩展到 ~30 个，支持：
- 分解策略: Boustrophedon 网格 / Sweep 扫描线
- 路径排序: Boustrophedon / Snake / Spiral / None
- 掉头规划: Direct / Dubins / Reeds-Shepp
- 边界策略: Closed / Open / Custom
- 各 Phase 优化独立开关

---

### 4.5 评估方法差异说明

| | 几何法（07-11 前主要使用） | 网格法（07-12 起主要使用） |
|------|------|------|
| 原理 | `swath总长 × 有效宽度 / 净面积` | 0.1m 分辨率逐点采样，`coverage_width/2` 判定 |
| 封顶 | `min(1.0, ...)` — 超 100% 截断 | 无封顶 |
| 精度 | 粗略，假设完美矩形 swath 覆盖 | 精确，反映真实路径几何 + headland 侵蚀 |
| 速度 | 毫秒级 | 小场景 <100ms，S3 约 1s（~45,000 格点，20-50x 加速后） |

**几何法报告的"100%"需谨慎解读**：在 S1/S5 等简单场景，swath 总面积略超净面积被 `min(1.0,...)` 截断为 100%，实际覆盖率（网格法）为 97-99%。这不是算法退化，是测量精度提升。

### 4.6 已知问题

**S4 窄走廊退化轨迹**：
```
07-11 (无sweep):       97.44%
07-12 (sweep):          95.00%  ← sweep 改变了 cell 布局
07-14 (sweep+Phase2-4): 90.08% (custom) / 95.15% (snake)
```
sweep 分解在 S4 窄走廊场景改变了 cell 布局，后续 Phase 2-4 优化未能完全弥补。Snake 模式 (95.15%) 优于 Custom (90.08%)，但均低于无 sweep 时的 97.44%。

**Phase 2C 重复触发 bug**：原始 benchmark 脚本的 publisher 重复发送 polygon 3 次，导致 Phase 2C 在同一个 planner 实例中被多次触发，每次累积 filler swaths 造成路径膨胀。S3_custom.log 中出现 39 份评估报告（非正常）。新管线修复：发布一次 polygon + 用 C++ 自动导出的最终路径 JSON。
