# F2C 全覆盖路径规划 — 算法评估框架

> 版本：v2.1  
> 日期：2026-07-11  
> 原则：**先定尺子，再量优劣。覆盖完整性是第一性原理。**

---

## 1. 评估目标

为 F2C 覆盖路径规划算法提供一套量化评估标准，使得：

- **同一算法不同参数**之间有可比性
- **不同优化方案**之间有客观的优劣排序
- **跨场景泛化能力**可以被验证
- **优化方向**有数据支撑，避免凭感觉改参数

---

## 2. 评估原则

| 优先级 | 原则 | 说明 |
|--------|------|------|
| 1 | **覆盖完整性第一** | 覆盖率不达标 = 方案无效，一票否决 |
| 2 | **泛用性优先** | 单个场景好不算好，多类场景都过关才算 |
| 3 | **效率在完整性之后** | 覆盖率达标后，才比较路径长短、转弯多少 |
| 4 | **可复现** | 同样的多边形 + 同样的参数 → 同样的结果 |

---

## 3. 评估指标体系（三层金字塔）

```
         ┌─────────────┐
         │  泛用性      │  ← 跨场景通过率
         │  (第三层)    │
        ┌┴─────────────┴┐
        │  效率指标      │  ← 路径长度/转弯/重叠/耗时
        │  (第二层)      │
       ┌┴───────────────┴┐
       │  覆盖完整性      │  ← 覆盖率 ≥ 99.5%
       │  (第一层)        │  ← 不达标 → 一票否决
       └─────────────────┘
```

### 3.1 第一层：覆盖完整性（硬门槛，一票否决）

| 指标 | 公式 | 阈值 | 判定 |
|------|------|------|------|
| 覆盖率 (Coverage Rate) | `实际覆盖面积 / 目标区域净面积 × 100%` | ≥ 99.5% | ✅ 合格 |
| | | 95.0% ~ 99.5% | ⚠️ 警告 |
| | | < 95.0% | ❌ 不合格 |

**目标区域净面积** = 多边形面积 - 孔洞面积

#### 覆盖率计算方法

##### 方法 A：网格栅格法（精确，用于最终评定）

```
1. 将多边形包围盒以 grid_resolution (建议 0.05m) 栅格化
2. 对每个网格单元：
   - 判断中心点是否在多边形内 && 不在任何孔洞内 → 属于"目标区域"
   - 判断中心点到路径任意段的最小距离 ≤ coverage_width/2 → 属于"已覆盖"
3. 覆盖率 = 已覆盖网格数 / 目标区域网格数 × 100%
4. 未覆盖区域可视化：标记所有属于目标但未被覆盖的网格
```

- **优点**：精确、可定位漏扫位置、可可视化  
- **缺点**：计算量大（0.05m 栅格，100m² ≈ 40,000 格）  
- **适用场景**：最终评估、方案对比
- **⚠️ 重要（v2.1 修正）**：目标区域使用**全图 cell 多边形**（含 headland 区域），而非侵蚀后的 `no_hl`。Headland 是路径的一部分，机器人在 headland 上行驶时也在覆盖，因此应纳入目标面积。用 `no_hl` 作目标会虚高覆盖率（分母变小）

##### 方法 B：几何估算法（快速，用于日常迭代）

```
覆盖率 ≈ min(1.0, (Σ swath_length × coverage_width - 重叠估算) / 净面积)
重叠估算 ≈ Σ(相邻swath之间的距离差 × swath长度)
```

- **优点**：计算快，适合频繁调参时实时反馈  
- **缺点**：不精确（未考虑 Headland 转弯区域的覆盖）  
- **适用场景**：日常开发调试

##### 评估时使用哪种？

| 场景 | 推荐方法 |
|------|----------|
| 日常调参/开发 | 方法 B（几何估算），实时反馈 |
| 最终对比/发布 | 方法 A（网格法），精确评定 |
| 覆盖率 < 95% 警告 | 自动触发方法 A 精确核验 |

### 3.2 第二层：效率指标（覆盖率达标后参与评分）

| # | 指标 | 英文 | 计算方式 | 优化方向 |
|---|------|------|----------|----------|
| 1 | **有效工作比** | Work Ratio | `Swath总长 / 路径总长` | ↑ 越高越好 |
| 2 | **路径总长** | Total Distance | 所有相邻路径点欧氏距离累加 (m) | ↓ 越短越好 |
| 3 | **转弯次数** | Turn Count | 方向变化 > 30° 的次数 | ↓ 越少越好 |
| 4 | **重叠率** | Overlap Rate | `重复覆盖面积 / 净面积` | ↓ 越低越好 |
| 5 | **规划耗时** | Planning Time | `planCoveragePath()` 执行时间 (ms) | ↓ 越快越好 |
| 6 | **路径平滑度** | Path Smoothness | 路径中各段曲率的均值 / 最大值 | ↓ 越小越平滑 |

#### 各指标详细说明

##### 3.2.1 有效工作比 (Work Ratio)

```
Work Ratio = Swath总长 / 路径总长
```

- Swath 总长：所有覆盖条带的直线长度之和（实际干活的路）
- 路径总长：Swath + Dubins 转弯 + Headland 过渡路径的总和
- 理想值：趋近于 1.0（意味着所有行驶都在干活，没有浪费）
- 实际典型值：0.7 ~ 0.9

**为什么用比值而不是绝对值？**  
因为不同多边形的面积不同，绝对路径长度无法跨场景比较。比值可以。

##### 3.2.2 路径总长 (Total Distance)

```
Total Distance = Σ distance(Path[i], Path[i+1])  for i in [0, N-2]
```

- 单位：米
- 包含所有路径段：Swath 直线 + Dubins 曲线 + Headland 连接弧
- 跨场景比较时，应除以 sqrt(净面积) 做归一化

##### 3.2.3 转弯次数 (Turn Count)

```
Turn Count = count of i where |Path[i+1].angle - Path[i].angle| > 30°
```

- 30° 阈值可配置
- Dubins 曲线的平滑过渡不会被计数（曲率连续变化）
- 仅在 Swath 端点处的 U 型掉头会被计为转弯

##### 3.2.4 重叠率 (Overlap Rate)

```
Overlap Rate = 重复覆盖面积 / 净面积
```

- 相邻 Swath 之间正常有少量重叠（确保无遗漏缝隙）
- 非正常重叠：Headland 区域被 Swath 从头扫到另一头导致的双层覆盖
- 理想：< 5%
- 可接受：5% ~ 15%
- 需优化：> 15%

##### 3.2.5 规划耗时 (Planning Time)

```
Planning Time = wall_clock_time at end of planCoveragePath - wall_clock_time at start
```

- 单位：毫秒
- 只计算算法执行时间，不含 ROS 通信、可视化发布等开销
- 对于"实时重规划"场景特别重要

##### 3.2.6 路径平滑度 (Path Smoothness)

```
Path Smoothness (avg) = 1/N × Σ curvature(Path[i])  for all path points
Path Smoothness (max) = max(curvature(Path[i]))    for all path points
```

- 曲率 = |Δ角度| / 位移距离（仅在有位移时定义）
- 最大曲率超过 `max_diff_curv` 意味着机器人实际无法跟踪该路径
- 这其实也是一个硬门槛：`max_curvature ≤ max_diff_curv`

### 3.3 第三层：泛用性（跨场景验证）

在 **6 类测试场景** 上都通过第一层（覆盖率 ≥ 99.5%），才算泛用性合格。

| 场景编号 | 类型 | 外边界 | 内部障碍 | 测试目的 | boundary_type |
|----------|------|--------|----------|----------|---------------|
| S1 | 空房间基准 | 20m×15m 规整矩形 | 无 | 基准性能 — 最简场景，算法底线 | closed |
| S2 | L 形房间 | 规整 L 形（常见户型） | 无 | 区域分解合理性 — Boustrophedon 对凹多边形 | closed |
| S3 | 带障碍房间 | 25m×25m 规整矩形 | 1 个多边形障碍物 | 障碍物下的避障与覆盖率保障 | closed |
| S4 | 狭长走廊 | 2m×20m 规整矩形 | 无 | 窄空间 swath 方向 + 掉头可行性极限 | closed |
| S5 | **两室一门连通** | 12m×8m 规整矩形，中部 1m 门洞 | 无 | ★ **门洞窄通道导航**：多边形自身收窄模拟房间门，测试跨房间覆盖路径 | closed |
| S6 | **两室一门+障碍** | 12m×8m，1m 门洞 + 1 个矩形障碍 | 1 个规整矩形（柜子） | 门洞导航 + 障碍避让组合测试 | closed |

> **2026-07-08 重新设计**：根据产品要求"外边界规整、多房间门连通"，重写了 S5/S6。
> - S5：多边形自身形状形成门洞（无 holes），测试窄通道导航能力
> - S6：在 S5 基础上加入规整障碍物，测试门洞+障碍组合
> - 全部场景 `closed` 硬边界，外边界均为规整几何形

**泛用性得分** = 通过率 = 覆盖率达标场景数 / 总场景数 × 100%

#### 边界策略测试矩阵

根据产品要求（2026-07-08），**绝大多数场景为硬边界 closed**。全部 6 场景默认 `closed`。`open` 仅作为交叉对比测试存在。

| 测试组 | 场景 | boundary_type | 验证目标 |
|--------|------|---------------|----------|
| **A 组：闭合基准（主力）** | S1~S6 全部 | `closed` | 所有 6 类场景下覆盖率 ≥ 99.5%，Swath 端点距边界 ≥ 0.2 m |
| **B 组：开放交叉对比** | S1, S5 | `open` | 取规整场景(S1)和复杂场景(S5)临时切换 open，验证覆盖率提升幅度 |
| **C 组：custom 手动** | S1 | `custom` + 手动 margin | 验证 custom 模式优先级高于 boundary_type 预设 |
| **D 组：障碍物不规则性** | S3, S6 | `closed` | 外边界的规整 holes 处理 vs 不规则 holes 的 swath 过滤效果 |

**边界策略对比指标**：

| 对比维度 | closed（主力，全部场景默认） | open（交叉对比用） | 判定标准 |
|----------|---------------------------|-------------------|----------|
| 覆盖率 | ≥ 99.0%（内缩留安全距离，牺牲少量边缘） | ≥ 99.5%（外伸补偿边缘） | closed 六场景均需达标 |
| Swath 端点距边界 | ≥ 0.2 m（不撞墙） | ≤ 0 m（可超出边界） | 端点坐标检查 |
| 重叠率 | 较低（内缩减少重叠） | 较高（外伸增加重叠） | open 重叠率 ≤ 15% |
| 路径总长 | 较短 | 较长（外伸拉长 swath） | 差异 ≤ 15% |

### 3.4 第四层：边界策略评分（独立维度）

因为闭合/开放的选择直接影响覆盖率、安全性和重叠率，需要独立对比。评分公式：

```
BoundaryScore = 0.5 × CoverageCompensation + 0.3 × SafetyMargin + 0.2 × OverlapControl
```

其中：
- `CoverageCompensation`：open 模式相比 closed 模式覆盖率的提升幅度（归一化）
- `SafetyMargin`：closed 模式下 swath 端点与边界的平均距离 / `swath_endpoint_shrink_distance`（越接近 1.0 越好）
- `OverlapControl`：`1.0 - overlap_rate`（两种模式下都要求重叠可控）

## 4. 综合评分公式

### 4.1 单场景评分

```
SingleScore = CoverageGate × EfficiencyScore
```

其中：

```
CoverageGate = coverage_rate ^ 10
```

- 覆盖率 100% → 1.000
- 覆盖率 99% → 0.904
- 覆盖率 98% → 0.817
- 覆盖率 95% → 0.599
- 覆盖率 90% → 0.349

**`^10` 的作用**：覆盖率差一点点分数就大幅下降，确保算法不敢以牺牲覆盖为代价换效率。

```
EfficiencyScore = 100 × (                    // direct 模式（零半径原地转，5 项）
    0.33 × WorkRatio_norm                    // 平滑度 5% 分配到工作比和路径长度
  + 0.27 × DistScore
  + 0.15 × TurnScore
  + 0.15 × OverlapScore
  + 0.10 × TimeScore
)

EfficiencyScore = 100 × (                    // dubins/reeds_shepp 模式（6 项）
    0.30 × WorkRatio_norm
  + 0.25 × DistScore
  + 0.15 × TurnScore
  + 0.15 × OverlapScore
  + 0.10 × TimeScore
  + 0.05 × SmoothScore                      // 仅非 direct 模式参与评分
)
```

> direct 模式去掉平滑度是因为差速驱动机器人可原地转，不存在"转弯不够平滑"的问题。重叠率和路径长度得分已根据 `swath_overlap_ratio` 自动调整，故意设的重叠不扣分。

各分项得分函数：

| 分项 | 得分函数 | 说明 |
|------|----------|------|
| `WorkRatio_norm` | `min(1.0, work_ratio)`（封顶防止因重叠>1.0溢出） | 越高越好 |
| `DistScore` | `exp(-total_dist / (10 × sqrt(area)))` | 路径越短分越高 |
| `TurnScore` | `exp(-turn_count / swath_count)` | 每次 swath 平均不到 1 个转弯最佳 |
| `OverlapScore` | `exp(-overlap_rate)`（指数衰减，>100%仍有区分度） | 重叠越低越好 |
| `TimeScore` | `1.0 - min(1.0, planning_time_ms / 5000)` | 5 秒以上得 0 分 |
| `SmoothScore` | `1.0 - min(1.0, avg_curvature / max_diff_curv)` | 曲率超 max_diff_curv 不得分 |

### 4.2 跨场景综合得分（6 场景加权平均）

#### 场景权重分配

不同场景的重要性不同——基准场景是"必须过"的底线，复杂场景考验泛化能力：

| 场景 | 权重 | 理由 |
|------|------|------|
| S1 空房间基准 | **20%** | 基准性能——这里翻车说明基本算法有问题 |
| S2 L 形房间 | **15%** | 常见户型，分解能力核心考察 |
| S3 带障碍房间 | **15%** | 障碍物避障基本功 |
| S4 狭长走廊 | **10%** | 极端几何，考验窄空间，但不常见 |
| S5 两室一门连通 | **25%** | ★ **核心场景**：门洞窄通道是多房间导航的本质挑战，最接近真实户型 |
| S6 两室一门+障碍 | **15%** | S5 基础上叠加障碍物，组合压力测试 |

> 权重合计 = 100%。S5+S6 合占 40%，覆盖了"门连通"这一核心产品需求。

#### 加权平均公式

```
WeightedAvgScore = Σ (SceneWeight_i × SingleScore_i)   for i in S1..S6
```

其中 `SingleScore_i` = 第 4.1 节的单场景得分。

#### 稳定性惩罚

除了加权平均，还要惩罚场景间得分方差过大（说明算法不稳定）：

```
VariancePenalty = standard_deviation(SingleScore_S1..S6) / 100

CrossSceneScore = WeightedAvgScore × (1 - VariancePenalty)
```

- 6 个场景得分都接近 → 方差小 → 惩罚小（算法稳定）
- 某些场景得分很高、某些很低 → 方差大 → 惩罚大（算法挑场景）

#### 最终综合得分

```
FinalScore = 0.55 × CrossSceneScore         // 跨场景泛用性（主维度）
           + 0.15 × BoundaryScore            // 边界策略合理性（第 3.4 节）
           + 0.30 × MinSingleScore           // 最差场景得分（短板惩罚）
```

**权重设计理由**：
- **55% 跨场景**：核心——算法在多种场景下都得好
- **15% 边界策略**：扫地机器人特有需求——知道什么时候该安全、什么时候该覆盖
- **30% 最差场景**：**短板效应**——木桶能装多少水取决于最短的板。一个场景得 30 分意味着算法在真实世界里遇到类似户型就会翻车，这不能容忍

> **为什么 MinSingleScore 权重高达 30%？**
>
> 均值容易被高分场景拉高，掩盖个别场景的严重失败。方差惩罚虽能部分补偿，但力度不够（标准差/100）。举例：
> - S1~S5 全部 92 分，S6 只有 35 分 → 均值 82.5，看起来不错
> - 但 S6 翻车说明算法面对多 disconnected 区域时几乎不可用
> - MinSingleScore = 35 → 直接贡献 0.30×35 = 10.5 到总分
> - 跨场景分也会被方差惩罚压低，双重保险确保"偏科"算法无法蒙混过关

#### 得分等级

| 等级 | 分数区间 | 含义 |
|------|----------|------|
| 🏆 S 级 | ≥ 90 | 优秀：所有场景覆盖率 ≥ 99.5%，效率指标均衡，可部署 |
| ✅ A 级 | 80 ~ 89 | 良好：大部分场景达标，个别场景需微调参数 |
| ⚠️ B 级 | 65 ~ 79 | 及格：基本功能可用，但效率或泛用性有明显短板 |
| ❌ C 级 | 50 ~ 64 | 不及格：多个场景覆盖率不达标，算法需要修复 |
| 🚫 D 级 | < 50 | 不可用：基础功能存在问题 |

#### 评分示例（假设数据）

```
场景得分：
  S1: 92.5 × 0.25 = 23.13
  S2: 88.3 × 0.20 = 17.66
  S3: 85.1 × 0.15 = 12.77
  S4: 79.2 × 0.10 =  7.92
  S5: 83.7 × 0.20 = 16.74
  S6: 76.8 × 0.10 =  7.68
  ─────────────────────────
  WeightedAvgScore = 85.90

  标准差 = 5.42
  VariancePenalty = 0.0542
  CrossSceneScore = 85.90 × (1 - 0.0542) = 81.24

  BoundaryScore = 88.0（假设边界策略表现良好）
  MinSingleScore = 76.8（S6 最低）

  FinalScore = 0.55 × 81.24 + 0.15 × 88.0 + 0.30 × 76.8
             = 44.68 + 13.20 + 23.04
             = 80.92  →  ✅ A 级

  解读：
  - 跨场景贡献 44.68 分（权重 55%）—— S4 和 S6 拖了后腿
  - 边界策略贡献 13.20 分（权重 15%）
  - 短板惩罚贡献 23.04 分（权重 30%）—— S6 的 76.8 分被放大检视
  - 如果 S6 从 76.8 提升到 85，总分可提升 0.30×8.2 = 2.46 分
```

## 8. 与优化方案的对应关系

每条优化方案对应评估体系中的哪几个指标：

| 优化方案 | 主要影响的指标 | 预期改善 |
|----------|---------------|----------|
| Swath 多角度选择 | Swath 数量 ↓ → 转弯次数 ↓、路径总长 ↓ | WorkRatio +5~15%, TurnScore +10~20% |
| 子区域面积预过滤 | 微小区域不再生成单条 swath → 规划耗时 ↓ | TimeScore +5~10% |
| RDP 路径简化 | 路径总长 ↓、规划耗时 ↓ | DistScore +3~8%, TimeScore +5~15% |
| RDP 转弯阈值 (`path_simplify_turn_threshold`) | 转弯检测敏感度 → 路径保真度 vs 简化率 | 阈值 ↑=更多简化/更多Dubins点被保护；阈值 ↓=更激进简化/可能丢Dubins曲线 |
| OR-Tools 求解器日志 (`ortools_exact_solve`) | 求解器诊断输出（不影响路径质量） | 仅影响调试效率，不参与评分 |
| 边界策略 (`boundary_type` + `boundary_coverage_margin` + `boundary_open_default_margin`) | 覆盖率、重叠率、路径总长 | closed: 覆盖率略降但安全；open: 覆盖率↑但可能越界 |
| **掉头可行性检查** | 路径可执行性（事后检测 + RViz 红色标记，direct 模式跳过） | 仅在 dubins/reeds_shepp 模式执行。利用 `PathState.type==TURN` 逐段计算曲率 vs `1/min_turning_radius`，超标时日志告警 + 评估报告展示 + RViz Marker 标红 |
| **Swath 重叠参数** | 覆盖率 + 重叠率评分 | `swath_overlap_ratio` 缩小 swath 间距防漏缝。评估重叠扣分已扣除预期值，不罚故意设的重叠 |
| **direct 模式** | 路径可执行性 | `turn_planner_type="direct"` 默认，标准 Dubins + PathPlanning 自动走 headland 环（不越界）。Dubins/Reeds-Shepp 保留可选 |
| **自适应 Headland + Cell 端点优化** | 覆盖率 + 越界控制 | 全局预扫描 + 逐 cell 局部调整避免窄通道堵死；每个 cell 首端点朝向全局中心减少跨 cell 长线连接 |
| **Swath 重叠参数** | 覆盖率 + 重叠评分 | `swath_overlap_ratio` 缩小间距防漏缝。评估公式（覆盖面积、重叠率、路径长度参考）均已适配有效间距 |
| Swath 空间预排序 | 路径总长 ↓（TSP 初始解更好） | DistScore +3~5% |

> **2026-07-08 更新**：OR-Tools 已确认由 Fields2Cover 内部集成，`ortools_exact_solve` 参数控制 `genRoute(show_log)` 而非求解器切换。评估时不需将其作为优化变量——它对路径结果无影响。

### 9.3 新增参数测试指南（2026-07-08）

#### `path_simplify_turn_threshold`（默认 0.15 rad ≈ 8.6°）

控制 RDP 转弯点检测的敏感度：
- **增大** (0.20~0.30)：更多方向变化被当作"直线"，RDP 更激进简化 → 路径点更少，但可能丢失 Dubins 曲线细节
- **减小** (0.05~0.10)：更敏感的转弯检测 → 更多段被保护，RDP 更保守 → 路径更精确但点数更多

**测试建议**：
```
# 敏感度对比测试（S1 基准场景）
for threshold in 0.05 0.10 0.15 0.20 0.30; do
    ros2 param set /polygon_planner_node path_simplify_turn_threshold $threshold
    ros2 service call /clear_coverage_planner_cache std_srvs/srv/Trigger
    # 重新发布多边形 → 记录路径点数和覆盖率
done
```

#### `boundary_open_default_margin`（默认 -0.3 m）

当 `boundary_type=open` 且 `boundary_coverage_margin=0` 时的保底延伸值：
- **负值更大** (-0.5)：Swath 更大幅度延伸到 Headland 中 → 覆盖率 ↑ 但重叠率 ↑
- **负值更小** (-0.1)：Swath 微微延伸 → 覆盖率略低但重叠少

**测试建议**：
```
# 开放边界延伸量对比（S1 基准场景，临时切为 open 交叉测试）
for margin in -0.1 -0.2 -0.3 -0.5; do
    ros2 param set /polygon_planner_node boundary_open_default_margin $margin
    ros2 param set /polygon_planner_node boundary_coverage_margin 0.0
    ros2 service call /clear_coverage_planner_cache std_srvs/srv/Trigger
    # 重新发布多边形 → 记录覆盖率和重叠率
done
```
