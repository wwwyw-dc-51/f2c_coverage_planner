### 1.2 产品约束 — 设计决策的边界条件 (Slide 3)

> 来源: `docs/产品需求待办.md`

**决定了什么能做、什么不能做**：

| 约束 | 确认内容 | 设计影响 |
|:---|:---|:---|
| 应用场景 | 室内扫地机器人，非农机 | 硬边界墙壁，覆盖优先于效率 |
| 平台 | C++ / 嵌入式 | 算法不能太重，不能依赖 GPU |
| 驱动方式 | 差速驱动，可原地转 | `turn_planner_type = "direct"` (零半径欧氏直连) |
| 倒车 | 不能倒车 | 若加运动学约束应选 Dubins（暂未启用） |
| 边界类型 | 绝大部分 closed 硬边界 | swath 端点必须内缩留安全距离 |
| 多房间 | 通过门洞连通 (1m 宽) | 核心场景：多边形收窄 (S5/S6) |
| 覆盖目标 | ≥ 99.5% (硬门槛，一票否决) | 覆盖率不达标 = 方案无效 |
| 重叠容忍 | ≤ 15% 可接受 | 效率分中重叠率参与评分 |
| **Swath 重叠** | **3% 故意重叠** | `swath_overlap_ratio = 0.03`，防定位误差漏缝 |
| 地图预处理 | 不需要 | 多层过滤 (filterTinyCells + 自适应 headland) 更精细 |

**3% Swath 重叠的设计理由**：
- 有效 swath 间距 = `coverage_width × (1 - 0.03)` = 0.90 × 0.97 = **0.873m**
- 来源: 语雀建议，防止定位误差导致相邻 swath 之间出现漏缝
- 评估时扣除计划重叠：`effective_width = cov_width × (1 - overlap_ratio)`，不因故意重叠扣分
- 这是一个"工程余量"——理论最优间距是 0.90m，但现实世界需要容错

### 1.4 故事一：从极角序到贪心出口驱动 (Slides 4-5)

**为什么改**：旧方案用固定极角序（绕孔洞质心的角度排序 cell），不考虑实际端点位置，导致跨场景长距离连接。

**做了什么**：
- `greedyCellOrder`: 从 C0 出口开始，贪心选最近未访问 cell 端点
- 同时决定：下一个 cell + 入口方向 + 内部覆盖方向 + 新出口
- 4 entry × 4 exit = 16 候选，DP 选最优链
- 有孔洞：圆形绕洞序 + 贪心方向反转；无孔洞：纯贪心
- 穿洞惩罚：+1000（后强化为 +1e9）
- 顺便修复：`touches_outer` 全局门控移除 → notched C0/C1 各 +1 边界填充线

**效果**：跨场景连接合理化，notched 由 3 条 27m 穿洞对角线 → 0

**取舍**：保留极角序作为孔洞场景的遍历序基础（DP 需要固定序），无孔洞才走纯贪心。这个混合策略一直保留到 v10.0。

### 1.5 故事二：Sweep 分解 — 最大的架构决定 (Slides 6-7)

**为什么改**：F2C 原生 Boustrophedon 面对孔洞产生 O(n²) 碎片

**做了什么**：沿 X 轴扫描线分解，遇到孔洞边界 Y 坐标切分条带
- **S3: 42→6 cells** (-86%)
- **S6: +3.2% 覆盖率**

**但产生了连锁反应** — 这是整个项目最深层的架构认知：
```
Sweep 分解 (替代 F2C Boustrophedon)
  → 对多边形朝向敏感          → Phase 4A 倾斜 sweep 补偿
  → 产生碎片化条带             → Cell 合并 (两轮安全合并) 补偿
  → 斜边三角间隙               → 斜边检测 + 独立角度补偿
  → 全局角度对部分 cell 不优   → Per-cell veto 补偿
  → Veto 改变方向后可能穿洞    → 1e9 硬约束补偿
```

**核心教训**：Sweep 打破了 F2C 的原生假设；每个断裂点需要一个补偿层；补偿层互相耦合；碰任何一点都可能触发连锁。这是一个"根变更→补偿链"的典型案例。

**取舍**：明知 Sweep 会引入碎片化，但它把问题从 O(n²) 降到 O(n)，这个收益值得后续 5 层补偿。如果当初知道需要 5 层补偿还会不会选 Sweep？可能还是会——碎片化比孔洞穿越好修。

### 1.6 故事三：穿洞之战 — 从软到硬 (Slides 8-9)

**第一回合 (07/13) — 后置打补丁（失败）**
- 方案 A: 检测 RDP 简化后的穿洞段，沿孔洞边界插入绕行点
- 方案 B: 连接层脆弱性检测 → segfault → 回退
- **教训**：后置修复药不对症，根因在排序阶段

**第二回合 (07/14) — Cell 连接反转**
- notched 3 条 27m 穿洞对角线 → 反转 swath 序 + 穿洞检测
- 覆盖率 97.3%→98.6%，得分 +26
- **部分解决**，但仍有场景穿了

**第三回合 (07/20) — 硬约束**
- 根因定位：DP 中 16 个 entry variant 全部穿洞时 +1000 惩罚不够
- 修复：`+1000` → `+1e9`。DP 存在非穿洞候选时绝不会选穿洞路径；全部穿洞时选 +1e9 路径，交给 `repairRouteConnectionsAroundHoles` 处理
- **8/8 hole_crossings = 0**
- 最小修复原则：只改连接惩罚，不动 veto 阈值，不加迭代修复

**总教训**：组合优化中的软约束在退化情况下失效——正常时 +1000 够用，但全部候选穿洞时 DP 没有"对"的选项可挑。

### 1.8 故事五：X-cut 的三次回滚 (Slides 12-13)

**问题**：N3 货架场景，孔洞条带经 `difference(holes)` 后产生 GDAL 不规则碎片，合并也救不回来

**三次尝试**：

| 迭代 | 方案 | N3 | 退化 |
|:---|:---|:---:|:---|
| 1: 全局 X-cut | 所有孔洞 X 边界加入分割线 | 100% | S7/N13/S3 退化 |
| 2: 条件触发 | Y重叠+X不重叠孔洞对才切 | 100% | S7/N13 仍退化 |
| 3: 更严条件 | ≥2×2 或 1×3 才切 | — | 老板喊停："这是面向测试场景编程" |

**三次全部回滚**

**正确做法 (Scheme 2)**：
- 只在孔洞条带**内部**做 X-cut（不是全局）
- 沿孔洞 X bounds 创建子矩形 → 各自 intersection → 无洞直接加，有洞回退 difference
- N3: 98.1%→99.97%, **+15.2 分**
- 21/21 GATE PASS

**教训**：
1. 让方案驱动，不要自行加东西。全局 X-cut 从未在约定计划中
2. 面向测试场景编程是坏味道
3. 分解与合并是一对——产生干净矩形而非 GDAL 不规则碎片才是正解

### 1.9 故事六：边界填充的进化 (Slide 14)

**起点**：S1 矩形顶部 swath 覆盖不到边界

**问题 2**：`touches_outer` 全局门控误杀贴边 cell → notched 漏填

### 1.10 故事七：C-space 的兴衰 (Slide 15)

**为什么引入**：想把碰撞安全做到规划层，机器人中心始终在安全区域内
**为什么退役 (v9.8)**：
- 配置膨胀——多了 C-space 专用参数和代码路径
- S7 被安全门拒绝（1 个路径点距边界 0.37499997m vs 要求 0.375m，差 2.88e-8m 的浮点误差）
- 不如在后处理做物理足迹检查——更精确、更解耦
- `traversability_enabled` 默认 `false`
**替代方案**：空隙分类器 (`classifyUncoveredGaps`)——区分物理不可达空隙和算法遗漏
**教训**：不是所有安全机制都要做到规划层。后处理安全检查 + 空隙分类 = 更干净的方案。

### 1.11 故事八：实车参数 — 实测颠覆假设 (Slide 16)

| 参数 | Demo 旧值 | 实测值 | 影响 |
|:---|:---:|:---:|:---|
| robot_width | 0.95m | **0.75m** | headland/swath 间距全部变化 |
| coverage_width | 0.45m | **0.90m** | swath 数减半，路径长大幅缩短 |

- **教训**：物理参数是算法的基础。先测准，再优化。反过来做就是空中楼阁。

### 2.1 测评驱动的优化闭环 (Slide 20)

**引入量化评分体系**：
```
CoverageGate = clamp((c - 0.90) / 0.095, 0, 1)³   // 90%=0, 99.5%=1
Score = CoverageGate × EfficiencyScore              // 工作比+距离+转弯+重叠+耗时
```

**为什么这样设计**：
- 覆盖率不达标 = 一票否决（gate=0）
- 99.5% 以上才满分——防止算法钻 99% 的空子
- 效率分在覆盖率合格后才参与比较——覆盖优先于效率

**batch_test_v2.sh**：21 场景 × 全自动，15 分钟出结果
**quick_dev_test.sh**：5 场景秒级反馈，日常迭代用
**n3_repeat_test.sh**：单场景方差验证

**案例**：N11 59% 回归 → Codex 静审 + Claude 代码审查 → 1 小时定位 → 修复
**如果没有这套体系**：N11 的 59% 可能几周后才被发现，届时已经不知道哪个改动引入的

### 2.2 工程建模的务实取舍 (Slide 21)

**三层模型，精度逐级递减**：
```
实际机器人 (0.90×0.80m + φ0.45m清洁轮×2)
        │
        ├→ 物理足迹层 (碰撞检测)    精度最高: 0.05m 采样
        │   逐姿态采样矩形机身+双轮几何，仅后处理安全检查，不参与规划
        │
        ├→ 覆盖模型层 (swath 生成)   精度中等: coverage_width=0.90
        │   假设清洁带为完美矩形，沿轨方向简化: 清洁长度=车长
        │
        └→ 拓扑模型层 (分解/合并)    精度最低: robot_width=0.75
            圆形包络近似，故意保守
```

**已知简化 & 为什么可以接受**：

| 简化 | 影响 | 为什么 OK |
|:---|:---|:---|
| 清洁长度=车长 | swath 起点存在未清洁死角 | 仅在端点，面积极小 (<0.1%) |
| 转弯覆盖未建模 | 转弯时扫过的面积不计入覆盖率 | **保守方向**，实际覆盖 ≥ 报告值 |
| 转弯碰撞已检测 | 安全不妥协，覆盖可保守 | 物理足迹逐姿态检查 |
| 矩形覆盖带 | 边缘实际有渐弱区 | 0.5 代原型验证管线，精细化留给产品化 |

**碰撞安全与拓扑规划参数分离**：
- 早期 `robot_width` 一人分饰六角（headland / 合并 / 窄通道 / 最小面积 / 自适应 headland / 边界偏移）
- 引入 `PhysicalFootprintParams` 后：碰撞安全用独立参数，`robot_width` 回归纯拓扑
- 解耦实验 (`feat/param-decouple-0721`) 试过直接拆 → 发现隐藏依赖 → 回滚 → 改从另一角度解决

### 2.3 Vibe Coding 的工作流纪律 (Slide 22-23)

**问题**：AI 写代码快，炸得也快。怎么把 vibe coding 变成有纪律的工程？

**纪律一：模块化解耦**
```
polygon_planner_node (ROS2)        ← 只管通信和调度
        ↓
PlannerCore::plan() (纯 C++)       ← 算法管线，零 ROS 依赖
   ├── decomposer       (~400 行)
   ├── swath_generator  (~600 行)
   ├── boundary_filler  (~500 行)
   ├── path_planner     (~600 行)
   └── coverage_evaluator (~700 行)
```
- 每个模块 <800 行，职责单一 → AI 能理解的小单元
- 改算法不动 ROS → 降低试错成本
- 每个模块可独立单元测试

**纪律二：快速验证通道**
- F2C Python 接口 → 不编译也能验证想法（秒级 vs 分钟级）
- `quick_dev_test.sh` 5 场景 → 日常迭代秒级反馈
- 全量 `batch_test_v2.sh` → 提交前 15 分钟跑完

**纪律三：AI+人都能读的数据界面**
- JSON 输出（路径坐标、swath 信息、连接路由）→ AI 精准定位问题
- 42 张可视化 PNG（coverage + cells）→ 人一眼看出好坏
- 连接 JSON 之前只存 `from_cell/to_cell` → 现在存 `route_waypoints`（1087 条全部含实际路径）

### 2.5 人机协作经验 (Slide 25)

**三人分工**：
- 老板：架构决策、参数配置、实测建模、场景设计
- Claude：架构分析、代码审查、数据对比、方案评估
- Codex：代码实现、可视化、连接优化、批测脚本

**交叉审查方法论**：
- 声明→证据→定量推演（拒绝"我觉得改好了"）
- 两个 AI 独立审查，不互相背书
- Codex 写代码 → Claude 验证 → 发现问题 → Codex 确认→修复 → Claude 再验证
- 案例：Codex 的 `mergeCollinearSwathsAcrossCells` N11 59% → Claude 1 小时审查定位 → 移除 → 恢复

**工作流一图**：
```
Windows 宿主机                       Ubuntu VM
Claude(分析)+Codex(实现)  ←ssh→  ROS2 + colcon + batch_test
        ↓                              ↓
  改代码 → sync → build → 批测 → 拉回报告+可视化
        ↓                              ↓
  交叉审查 ←────────── 21 PNG + JSON ──┘
        ↓
  决策: 接受/回滚/继续
```

**核心原则**：重复性工作给工具，判断力留给自己。AI 是工具不是答案。
