# Codex Brief — 2026-07-22 脆弱性分析与分解优化

## 背景

改 robot 尺寸参数会导致算法崩溃 → 怀疑优化栈太脆弱 → 做了一次从项目 Day 1 到现在的全量考古。

## 核心发现：补偿链

```
Sweep 分解 (根改动，替代 F2C Boustrophedon 网格)
  ├── 对多边形方向敏感 ──→ Phase 4A 倾斜 Sweep 补偿
  ├── 产生碎片条带 ────→ Cell 合并补偿
  ├── 斜边三角缝隙 ────→ 斜边检测 + 独立角度补偿
  ├── 全局角度不适用 ──→ per-cell veto 补偿
  └── veto 改变方向 ────→ 1e9 穿洞硬约束补偿
```

**本质**：Sweep 打破 F2C 原生假设 → 每个打破点需要补偿 → 补偿层互相耦合 → 改动一点触发连锁反应。

同时 `robot_width` 有 6 重语义（headland 侵蚀/合并阈值/窄通道检测/min cell 面积/自适应 headland/边界补线偏移），`param-decouple` 实验证明存在未识别的隐藏依赖。

## Sweep 分解原理

- X 方向：压缩为 [min, max]，全宽条带不切割
- Y 方向：在外环顶点 + 孔洞顶点处画水平切割线
- 每个条带 = `intersection(全宽矩形, 多边形)`
- 含孔洞的条带 → `difference(孔洞)` → 碎片

**优势**：规整矩形 1 cell，S3 从 42→7 cells
**弱点**：密集孔洞场景（货架 4×3 网格）→ 孔洞行条带被 difference 炸成碎片 → cell 合并无法吸收（碎片被孔洞隔离）→ N3 仅 98.1%

## 本轮改动（方案 1+2）

### 方案 2：「有洞条带沿孔洞 X 边界拆子矩形」

**之前**：含孔洞的条带 = `difference(孔洞)` → GDAL 产生的不规则碎片
**之后**：含孔洞的条带 → 收集孔洞 X 边界 → 在边界之间创建子矩形 → `intersection(子矩形, 多边形)` → 对齐孔洞的干净矩形（通常无洞，若有洞回退 difference）

```
之前：全宽条带 + difference → [碎片A][洞][碎片B][洞][碎片C]
之后：子矩形逐个求交   → [矩形A][洞][矩形B][洞][矩形C]
      矩形A/B/C 是干净几何，不经过 GDAL difference
```

关键：「只切带洞块」——无孔洞的条带（入口/gap/出口）保持完整，不受影响。

### 方案 1：「保守垂直合并兜底」

在两轮合并（H-merge + V-merge）后再跑一次 `mergeAdjacentSweepStrips`，捕获方案 2 子矩形拆分后新产生的同 X 跨度相邻 cell。

## 走过的弯路（已回退）

最初误把方案 2 实现为"分解时全局加 X 切割"（给 xs 数组加孔洞 X 边界），导致：
- S7：9 cells → 41 cells（孔洞不规则，X 切割产生大量无法合并的碎片）
- 试图加条件过滤（≥2行×≥2列 / ≥3列）→ "面向测试编程"
- 第三轮合并基本是 no-op，没产生实质效果

**教训**：X 切割应该在"有洞条带内部"做，不是全局网格。

## 待思考的问题

1. **方案 1（第三轮 V-merge）是否有实质效果？** 初步分析认为它大部分场景是 no-op（垂直合并后 cell 变高，但不产生新的合并机会）。需要实际 cell 数对比验证。

2. **方案 2 的覆盖率比方案 1+2 联合的 V1（100% N3）略低（99.97% vs 100%）**。V1 的全局 X 切割 + 无条件合并在某些场景更激进，但副作用太大（S7 爆炸）。当前方案更保守但覆盖略低。

3. **合并能力是否足够？** 方案 2 产生干净子矩形后，mergeCellsWithSimilarDirection 的邻接阈值 `2*coverage_width=1.8m` 对 2m 宽货架恰好不够 → 同条带碎片无法水平合并。是否需要在同 Y 条带内放宽合并条件？

4. **robot_width 解耦** — 6 重语义需要理清，`param-decouple` 分支的实验结果可以参考。
